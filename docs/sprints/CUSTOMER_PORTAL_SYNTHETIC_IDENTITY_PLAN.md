# Customer Portal Synthetic Test Identity Plan

## Date and platform layer

Prepared July 14, 2026 for the **Customer Portal** platform layer.

## Status

**Planned; identities not provisioned; authentication not production-ready.** The current certification state has removed the earlier production-hardening verification blocker. The production-security migration remains Applied and immutable and must not be repeated. This plan does not authorize SQL execution, automatic account creation, production customer data, or use of a production customer identity.

## Environment gate

Provision only after a human approves the target environment, exact package revision, operators, test window, evidence retention, trusted provisioning paths, and cleanup owner. The preferred target is an isolated non-production Supabase project or branch carrying the same Applied schema. Provisioning in a production project requires separate explicit approval and is outside this plan.

All email aliases below use the reserved `.test` namespace. They are fictional labels, not deliverable public mailboxes. Password recovery requires an approved private test SMTP sink that accepts the namespace without public delivery. If that sink is unavailable, stop; do not substitute a real person's email.

## Supabase Auth records required

Exactly eight Supabase Auth users are required: Owner A, Owner B, A1, A2, A3, A4, A5, and B1. A5 is one Auth record shared by its two permitted customer profiles, not two Auth users. All eight emails must be confirmed through the approved operator process. UUIDs, passwords, recovery links, access tokens, refresh tokens, and raw metadata must remain secret.

This exact eight-user structure covers the requested 15 authentication scenarios only. It does not add the separate portal-only no-profile identity or intentionally mixed owner/customer and member/customer principals needed for broader database/RLS denial testing. Those fixtures are out of scope, remain unprovisioned, and require a separate human-approved security test plan; session-isolation scenario 14 must not be reported as mixed-principal database evidence.

Owner A and Owner B are workspace identities. A1 through A5 and B1 are portal-only identities. Do not add a customer identity as an owner or member of any tenant, including an unrelated tenant. Do not create a service-role “test user”; service-role authority is server-only and never enters the browser.

Both A5 profiles must be active, and each tenant must have distinguishable `customer_portal_branding` plus harmless customer-visible fixture data. If either profile is invited or suspended, the multi-profile selector is not validly exercised.

Each redeemable invitation must also satisfy `expires_at > statement_timestamp()` when the trusted redemption occurs. The raw high-entropy token is never stored in the database or documentation. A4's invitation remains unaccepted and its profile remains `invited`. Do not call redemption for A4. Redemption of a suspended A3 profile must be rejected; create the active profile through redemption first, then suspend it through the approved trusted administration path.

## Secure human provisioning sequence

1. A human confirms the current approved schema/security state and records that the Applied migration will not be altered or repeated.
2. A human approves the isolated non-production environment, package revision, two fictional tenants, all eight aliases, private mail sink, authorized operators, test dates, evidence rules, and cleanup ownership.
3. The operator follows the fixed order in the package: Businesses, Owners, Invitations, Auth Users, Customer Profiles, Documents, Appointments, Messages, then Support Requests.
4. Businesses are created first without an owner UUID; Owner A and Owner B are then created as Auth users and linked only to their matching business. Owners receive no portal profile or invitation.
5. The approved trusted server creates the seven portal invitations before customer Auth users. Supabase Auth **Send invitation** is not a substitute.
6. The operator creates A1, A2, A3, A4, A5, and B1 in Auth, confirms only their private test-mailbox aliases, sets exact full-name metadata before redemption, and verifies that none is an owner or member anywhere.
7. The trusted server redeems A1, A2, A3, A5-A, A5-B, and B1 through the service-role-only function. The raw token and service key never enter the browser. A3 is then suspended; A4 is created invited and remains unredeemed through approved privileged administration.
8. The operator creates the bounded customer-visible matrix, verifies exact counts, tuple isolation, and private Storage, delivers credentials through the secret procedure, and opens the approved test window.

## Credential-handling procedure

- Generate a unique high-entropy temporary password for each Auth identity in the approved password manager or secret-management system. Never reuse a password across identities.
- Share each credential through an expiring, access-controlled secret link to the named tester. Do not place it in Git, documentation, issue trackers, chat, email bodies, screenshots, terminal history, CI variables, or test output.
- Store the private mail-sink credentials separately from Auth credentials. A2 recovery messages must stay in that sink; never forward or screenshot a recovery URL.
- Browser/network evidence may record timestamps, response classes, redirects, and alias labels only. Redact cookies, authorization headers, query codes, tokens, UUIDs, and signed URLs.
- Rotate A2's password through the recovery scenario. Rotate every remaining password after the test window, then revoke sessions before deletion.
- Limit service-role access to the trusted provisioning operator. Never deliver the service key to a tester or browser and never use it as an identity.

## Cleanup and revocation procedure

1. Close the test window and revoke all active/refresh sessions for the eight Auth users through the approved Auth administration path.
2. Capture sanitized pass/fail evidence and unresolved defects without credentials, UUIDs, tokens, recovery links, signed URLs, or customer content.
3. Remove customer-visible child fixtures through the trusted administration service in dependency-safe order: acknowledgements, messages, requests, appointments, documents/storage objects, then profile-linked records.
4. Remove the seven profiles, seven invitations, synthetic branding, and any other Tenant A/B portal fixtures. Remove owner/member links and the two synthetic business records through the approved cleanup path.
5. Delete or disable all eight Auth users only after database ownership/link dependencies are cleared. Do not rely silently on cascades as the cleanup record.
6. Purge test messages from the private mail sink and destroy password-manager entries, raw invitation material, recovery links, and local evidence containing secrets.
7. Clear both browser storage keys, selected-profile local storage, cookies, service workers, and test browser profiles on every tester device.
8. Run the approved read-only integrity/verification checks manually, confirm no reserved aliases or synthetic tenant records remain, and record cleanup approver, date, result, and exceptions.
9. If cleanup requires a schema change or no approved data-removal path exists, stop and start the mandatory database workflow. Do not run ad hoc SQL or rerun the Applied migration.

## Scenario-to-identity matrix

| # | Authentication scenario | Identity | Required execution and expected result |
| --- | --- | --- | --- |
| 1 | Valid customer login | A1 | Sign in at `/portal/login`; reach only Tenant A |
| 2 | Invalid password | A1 alias with a deliberately incorrect value | Reject authentication and create no portal session; never record the attempted value |
| 3 | Logout | A1 | Local portal logout clears the portal session and returns to login |
| 4 | Protected route after logout | A1 immediately after #3 | `/portal/messages` redirects to `/portal/login` and reveals no stale data |
| 5 | Browser refresh with active session | A1 | Full refresh preserves the active session and Tenant A context |
| 6 | Token refresh | A1 | Observe automatic refresh using an approved lifetime; record timing/status only, never token contents |
| 7 | Expired-session handling | B1 | Revoke B1's refresh session through the approved Auth control, then wait for the already-issued access JWT to expire or force the next approved refresh boundary. Confirm refresh fails and the portal returns safely to login without stale data. Revocation is not assumed to invalidate an issued access JWT immediately, and clearing local storage is not equivalent |
| 8 | Password recovery | A2 | Send to the private sink, use the valid recovery flow, set a new test password, and sign in |
| 9 | Recovery callback failure | A2 | Reuse/expire an issued link or use a malformed callback; reject safely without session or password mutation |
| 10 | Suspended customer | A3 | Auth may succeed, but zero active RLS-visible profiles denies portal access |
| 11 | Invited but inactive customer | A4 | Auth may succeed, but `invited` is not active and portal access is denied |
| 12 | Multiple business profiles | A5 | Clear remembered selection, show both authorized profiles, switch A to B, persist choice, and prove no cross-tenant data mixing |
| 13 | Multi-tab session propagation | A1 | Two same-origin tabs consistently reflect sign-in, refresh, and local logout changes without exposing token contents |
| 14 | Portal/Business Workspace isolation | A1 + Owner A | Sign A1 into `/portal/login` and Owner A into `/auth/login` in one browser profile; sessions coexist under separate keys, portal logout leaves workspace session intact, and A1 cannot satisfy workspace authorization. This tests two-principal session isolation, not mixed-principal database denial |
| 15 | Demo creates no Supabase client | No identity; clean development/test browser context | Load `/portal/demo`; create no Supabase Auth request, portal/workspace session, or auth-storage entry |

Recommended order: #15 in a clean context; A1 runs #2, #1, #5, #6, #13, #3, #4; A2 runs #8 then #9; B1 runs #7; A3 and A4 run #10 and #11; A5 runs #12; A1 plus Owner A run #14 in a fresh context. Owner B remains the Tenant B control owner; optionally mirror #14 with B1 and Owner B under separate approval.
